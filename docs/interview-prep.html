The Problem & Solution

What is ScopeWarden?

A guardrail system that ensures AI coding agents stay within the bounds of their declared mission.

🎯

Mission Scoping

Agents declare what they're working on. Every file change is evaluated against that mission to detect scope creep.

🛡

Policy Enforcement

Deterministic rules block changes to sensitive files (.env, secrets, keys). No LLM can override blocked patterns — hard stops.

📊

Live Audit Cockpit

A 5-mode terminal UI for real-time monitoring: Review, Chat, Dashboard, Sessions, and Live file watching.

🤖

Agent Agnostic

Works with Claude Code, Codex, Gemini, Cursor — any agent that can execute shell commands. Zero lock-in.

⚖️

LLM Judge

Optional AI-powered assessment of whether changes match the mission. Supports 5 providers including local Ollama.

📝

Session Tracking

Every agent session is recorded with timestamps, check results, and files changed — full audit trail.

🎤 Elevator Pitch: "ScopeWarden is a scope firewall I built in Rust that sits between AI coding agents and your codebase. It combines deterministic policy enforcement — like blocking .env files — with an optional LLM judge that evaluates whether changes match the declared mission. It's agent-agnostic by design: thin configuration adapters for each AI agent, but the core CLI is completely decoupled. Think of it as a security + governance layer for the emerging AI-assisted development workflow."

Policy Engine

Deterministic rule enforcement — the security core of ScopeWarden

Core Concept

The policy engine is deterministic and non-overridable. Unlike the LLM judge, policy verdicts cannot be influenced by prompt injection or model hallucination. This is a deliberate security architecture choice.

Key Types

Rust 
pub enum PolicyVerdict {
    Blocked,    // Hard stop — file matches a blocked pattern
    Warned,     // Soft warning — matches a warn pattern
    Allowed,    // No policy match — proceed
}

pub struct PolicyEngine {
    blocked_patterns: Vec<glob::Pattern>,
    warn_patterns:    Vec<glob::Pattern>,
    max_files:        usize,
    max_lines:        usize,
}

How It Works

1
config.rs → PolicyEngine::new()
Load scopewarden.yaml, compile glob patterns for blocked & warn lists into glob::Pattern instances
2
policy.rs → evaluate_file()
For each changed file, test path against all blocked patterns first (short-circuit), then warn patterns → PolicyVerdict
3
policy.rs → evaluate_changeset()
Evaluate all changed files, aggregate into PolicyReport with counts per verdict
4
policy.rs → check_limits()
Compare total file count and line count against max_files_changed and max_lines_changed thresholds

Configuration

YAML 
policy:
  blocked:
    - ".env"              # exact match
    - "**/.env.*"         # glob — any .env.* in any dir
    - "**/secrets/**"     # entire secrets directory tree
    - "**/*.pem"          # all PEM certificates
    - "**/*.key"          # all private keys
  warn:
    - "package-lock.json"
    - "yarn.lock"
    - "Cargo.lock"
  max_files_changed: 20
  max_lines_changed: 800

Scope Labels

BadgeLabelMeaningAction
● EXPECTEDExpectedFile matches the active mission scopeNo action needed
● SUSPICIOUSSuspiciousChanged but no mission rule matchesReview recommended
● BLOCKEDBlockedMatched a blocked policy pathHard stop — cannot proceed
● IGNOREDIgnoredClean, stale, or explicitly excludedSkipped
🎤 Interview talking point: "The policy engine is intentionally non-Turing-complete. By using declarative YAML glob patterns instead of a scripting language, I ensure that policies are auditable, version-controllable, and cannot contain logic bombs. Blocked patterns are evaluated deterministically — there's no LLM in the loop for security-critical decisions."

Git Diff Analysis

Pipeline-based change detection and classification

Data Flow Pipeline

The diff module implements a clean pipeline: Git Diff → Parse → Classify → Report

Rust 
pub struct ChangedFile {
    pub path:          String,
    pub status:        FileStatus,      // Added | Modified | Deleted | Renamed
    pub lines_added:   usize,
    pub lines_deleted: usize,
}

pub enum ScopeLabel {
    Expected,    // Mission match + policy allowed
    Suspicious,  // No mission match, policy allowed
    Blocked,     // Policy blocked (overrides everything)
    Ignored,     // Excluded from analysis
}

Classification Logic

1
git2 → get_changed_files()
Use libgit2 binding to compare HEAD vs. working tree — no shell subprocess spawning
2
For each file → policy.evaluate_file()
Check against blocked patterns first (short-circuit on match), then warn patterns
3
Match against mission.expected_paths
If policy allows, check whether the file path matches any glob in the mission's expected paths
4
Combine → ScopeLabel
Blocked overrides all. Expected = policy OK + mission match. Suspicious = policy OK + no mission match.
🎤 Interview talking point: "I chose git2 (libgit2 Rust binding) over shelling out to git for three reasons: no process spawning overhead, structured data access instead of string parsing, and cross-platform reliability. The tradeoff is a larger binary and a C dependency, but for a tool that runs on every commit check, the performance gain matters."

LLM Judge System

Multi-provider AI assessment with structured output

Architecture: Strategy Pattern

The judge uses the Strategy pattern — a JudgeProvider enum dispatches to provider-specific API implementations. This keeps the core logic clean while supporting 5 different LLM backends.

Rust 
pub enum JudgeProvider {
    Ollama,     // Local, private — no API key needed
    Claude,     // requires ANTHROPIC_API_KEY
    OpenAi,     // requires OPENAI_API_KEY
    Gemini,     // requires GEMINI_API_KEY
    OpenRouter, // requires OPENROUTER_API_KEY
}

pub struct JudgeVerdict {
    pub file:       String,
    pub relevant:   bool,
    pub confidence: f64,     // 0.0 to 1.0
    pub reasoning:  String,
}

Provider Architecture

🏠

Ollama (Local)

Default provider. Runs locally — zero data leaves the machine. Best for sensitive codebases. No API key required.

☁️

Cloud Providers

Claude, OpenAI, Gemini — higher quality judgments but requires API keys and sends code to external services.

🌐

OpenRouter

Meta-provider — access to hundreds of models through one API. Good for experimentation and comparison.

Deterministic + LLM Hybrid Design

Critical design decision: the judge is an additional layer, not a replacement for the policy engine.

AspectPolicy EngineLLM Judge
NatureDeterministicProbabilistic
Override-ableNo — hard enforcementAdvisory only
SpeedMicroseconds (glob matching)Seconds (API calls)
AccuracyPerfect for known patternsBest for semantic judgment
Attack surfaceNone (no LLM)Prompt injection possible
Offline capableYesOllama only
🎤 Interview talking point: "This is a defense-in-depth design. The deterministic policy engine provides the security floor — it's fast, offline-capable, and immune to prompt injection. The LLM judge adds semantic intelligence on top: it can understand whether a change to utils.rs is related to a 'fix login bug' mission even if the file isn't in the expected paths list. But it can never override a BLOCKED verdict."

Git Hooks Enforcement

Automatic scope compliance — mandatory enforcement before every commit

Why This Exists

Without hooks, agents run scopewarden check voluntarily. An agent could simply skip the check and commit blocked files. The pre-commit hook makes enforcement mandatory — the commit physically cannot proceed if blocked files exist.

Architecture Decision: Shell Script over Binary Hook

✅ Chose: Shell Script

Works on any system with /bin/sh. Users can read and modify the hook. Self-referential — calls back to the same scopewarden binary.

❌ Rejected: Binary Hook

Would require compiling a separate binary, distributing it per-platform, and keeping it version-synced with the main CLI. Way too complex.

❌ Rejected: husky/lint-staged

Node.js dependency. ScopeWarden users may not have Node installed. A Rust tool shouldn't depend on the JavaScript ecosystem for enforcement.

Key Design: Non-Destructive Install

  • Backup — existing pre-commit hooks are copied to .git/hooks/pre-commit.bak before overwriting
  • Marker detection — uses # ScopeWarden pre-commit hook comment to identify its own hooks vs third-party
  • Clean uninstall — removes only ScopeWarden's hook, restores backup if one exists
  • Escape hatch — git commit --no-verify always works for emergencies
🎤 Interview talking point: "The hook enforces policy at the git boundary — the last line of defense before code enters the repository. I deliberately chose a shell script over a compiled binary hook because transparency matters for a security tool. Users should be able to read cat .git/hooks/pre-commit and understand exactly what's being enforced. The non-destructive install pattern — backup, marker detection, clean uninstall — ensures we're a good citizen alongside other git hooks like prettier or eslint."

Chat System

Persistent conversational scope review with LLM integration

Why Not Just Binary Verdicts?

The LLM judge returns Matches | Drift | Unknown — but developers need to understand why. The chat system lets users have a conversation: "Why did you flag utils/crypto.rs? The login bug fix needs password hashing changes." This turns the judge from a black box into an interactive advisor.

Data Architecture

Filesystem 
.scopewarden/chats/
├── index.json          # Master index of all chat sessions
├── <ULID>/
│   ├── context.json    # Provider, model, repo root
│   └── messages.jsonl  # Append-only message log
└── archive/
    └── <ULID>/          # Soft-deleted chats (recoverable)

Why These Persistence Choices

📝 JSONL for Messages

Crash-safe: Each line is self-contained. A crash mid-write loses at most one entry. Stream-friendly: Can be tailed with tail -f and processed with jq.

🗂 Soft Delete → Archive

Recoverability: Prevents accidental data loss. chat delete moves to archive; chat restore brings it back; chat purge --yes permanently removes.

🎤 Interview talking point: "I implemented a three-tier deletion model — active, archived, purged — because for a governance tool, audit history matters. The soft-delete pattern means accidental deletions are always recoverable. I chose JSONL over SQLite for messages because it's crash-safe (append-only), human-readable (cat messages.jsonl | jq), and has zero native library dependencies. This is the same format used by structured logging systems like Bunyan."

Agent Integration & MCP

How ScopeWarden achieves agent-agnosticism across Claude, Codex, Gemini, and Cursor

Design Philosophy

ScopeWarden uses a thin adapter layer approach. The core CLI is completely agent-agnostic — it doesn't know or care which AI agent is running. Each agent gets a configuration file in its native format that teaches it how to use ScopeWarden.

Integration Architecture

Claude Code

CLAUDE.md + .claude/commands/

  • CLAUDE.md — system prompt with mandatory rules
  • Slash commands — scope-init, scope-start, scope-check, scope-watch, scope-attach
  • .claude-plugin/manifest.json — distributable plugin package

OpenAI Codex

.codex/instructions.md

  • Single instructions file with workflow rules
  • Same mandatory checkpoints as Claude
  • Adapted to Codex's instruction format

Google Gemini

AGENTS.md

  • Uses Gemini's AGENTS.md convention
  • Quick reference tables for commands and shortcuts
  • Judge provider configuration docs

Cursor

.cursor/rules/scopewarden.mdc

  • Uses Cursor's MDC (Markdown Config) format
  • alwaysApply: true — rules always active
  • Same workflow integration rules

Agent Workflow Integration

Every agent follows the same workflow, regardless of its configuration format:

1
Before starting work
scopewarden status — Check if there's an active mission, verify the environment is ready
2
Start mission
scopewarden start "your mission" — Declare what the agent is about to do
3
During work (optional)
scopewarden watch — Live TUI cockpit for monitoring in real-time
4
Before finishing
scopewarden check — Full policy check + scope audit. Must pass before commit.
5
Before committing
scopewarden diff --problems — Show only suspicious/blocked files for final review

MCP Relationship

ScopeWarden does not implement MCP (Model Context Protocol) directly. Instead, it's a CLI tool that MCP-enabled agents can invoke via their command execution capabilities. The relationship is:

  • AI agents use MCP to access tools (file read, command execution, etc.)
  • ScopeWarden is one such tool — invoked via shell commands
  • The config adapters teach agents when and how to invoke ScopeWarden
🎤 Interview talking point: "I designed the integration layer as thin adapters rather than building agent-specific code into the core. Adding support for a new AI agent is just creating a config file — zero Rust code changes. This is the Open/Closed Principle in practice: open for extension (new agents), closed for modification (core CLI). The CLI-first design means any tool that can run a shell command can use ScopeWarden, making it future-proof as new agents emerge."

Internal Architecture as "Pluggable" Modules

While the plugins/ directory is reserved for future external plugins, the core system itself is designed with pluggable components:

🔌 Pluggable Judge Providers

5 LLM backends via JudgeProvider enum. Adding a new provider = adding an enum variant + API call function. Strategy pattern.

🎨 Pluggable TUI Themes

5 built-in themes (scopewarden, codex, claude, openclaw, high-contrast). Theme struct with color fields — easy to extend.

📋 Pluggable Agent Configs

Each agent's config is an independent file in its own format. No coupling between agents. Adding a new agent = new config file.

📏 Pluggable Policy Rules

Declarative YAML patterns. Users can customize blocked/warn lists and limits without touching code. Version-controlled alongside the project.

Error Handling Strategy

  • Rust's Result<T> with anyhow crate for ergonomic error propagation
  • ? operator for clean error bubbling — no manual unwrap calls
  • anyhow::bail!() for early returns with descriptive messages
  • anyhow::Context for adding context to errors ("failed to load config")
  • No panics in normal code paths — graceful degradation
  • Exit codes: 0 = success, 1 = policy violations found, 2 = configuration error
Engineering Judgment · Why This Architecture

Architecture Decisions & Tradeoffs

Every choice has a WHY — here's how to defend each decision in an interview

🦀 Rust over Python/Go

Chose: Performance, memory safety, single-binary distribution.
Over: Larger ecosystem (Python), faster compile times (Go), more contributors.
Why: A scope firewall runs on every commit check — it must be fast and reliable. Single binary = zero runtime dependency. No virtualenvs, no node_modules.
Tradeoff: Slower development, steeper learning curve, smaller contributor pool.

📚 git2 over shell git

Chose: In-process library, typed Rust structs, no process spawning overhead.
Over: Smaller binary, simpler integration, git CLI familiarity.
Why: The TUI refreshes every 150ms — subprocess spawning would be too slow. Parsing git diff output is fragile and locale-dependent.
Tradeoff: Larger binary (libgit2 is a C library), C build dependency. Used default-features = false to disable SSH/HTTPS and minimize attack surface.

📝 YAML Policy over Code Policy

Chose: Declarative, auditable, version-controllable, non-Turing-complete.
Over: Flexible scripting (Lua, Starlark), conditional logic, dynamic rules.
Why: Security invariant — policies can't contain logic bombs. What you see is what you get. Auditors can review glob patterns without understanding a programming language.
Tradeoff: Can't express "block .env except in test environments" without code. Accepted limitation for security.

🖥 TUI over Web UI

Chose: Zero additional dependencies, terminal-native, fits developer workflow.
Over: Richer visuals, better data visualization, mobile access, collaborative viewing.
Why: Target users are developers working in terminals alongside AI agents. A web UI adds a server dependency and opens a port.
Tradeoff: Limited to terminal capabilities. No graphs, no images, no drag-and-drop.

📄 JSON Files over SQLite

Chose: Human-readable, git-friendly, no additional native library dependency.
Over: Query capabilities, indexing, concurrent access, scalability.
Why: Session data is small. JSON files are inspectable with cat. No SQLite = simpler cross-compilation (critical for musl static builds).
Tradeoff: No query capability. As sessions grow into thousands, would need indexing.

🤖 Enum Dispatch over Trait Objects

Chose: JudgeProvider enum with exhaustive match — compile-time guarantees.
Over: dyn JudgeProvider trait objects — runtime polymorphism, open extension.
Why: Closed set of 6 providers. Adding one = adding an enum variant. Exhaustive match catches every missing case at compile time. No vtable overhead.
Tradeoff: Can't add providers without code changes. If providers grew to 20+, would refactor to traits.

🆔 ULID over UUID

Chose: Monotonically sortable unique IDs — ls .scopewarden/ shows sessions in chronological order.
Over: UUID v4 — universally recognized, library support everywhere.
Why: ULIDs embed a timestamp prefix. Sessions naturally sort by creation time without parsing metadata. Better for auditing and debugging.
Tradeoff: Less common than UUID; some tools don't understand the format.

📦 Pure Binary Crate (no lib.rs)

Chose: Flat module design, no public API contract to maintain.
Over: lib+bin split for reusability, programmatic embedding, IDE plugins.
Why: ScopeWarden is an end-user CLI tool. Integration tests via assert_cmd test the actual binary. No downstream consumers need a library API.
Tradeoff: Cannot be embedded as a library. Would need refactoring for IDE plugin scenarios.

🎯 GlobSet DFA over Individual Globs

Chose: globset::GlobSet — compiles all patterns into a single deterministic finite automaton.
Over: Individual glob::Pattern matching — simpler but O(patterns × paths).
Why: O(path_length) matching regardless of pattern count. Uses same syntax as .gitignore — zero cognitive load for developers.
Tradeoff: Upfront compilation cost, more complex error handling for malformed patterns.

🔒 rustls-tls over OpenSSL

Chose: Pure Rust TLS implementation — no native C library dependency.
Over: OpenSSL — more widely tested, hardware acceleration support.
Why: Eliminates entire class of native library vulnerabilities. Simpler cross-compilation. No OpenSSL version conflicts.
Tradeoff: Slightly slower TLS handshakes. No hardware HSM support.

🎤 Interview talking point: "I can justify every dependency in my Cargo.toml. Each one solves a specific problem that I evaluated alternatives for. For example, I chose globset over individual glob::Pattern matching because GlobSet compiles all patterns into a single DFA — that's O(path_length) matching regardless of how many patterns exist. I chose rustls-tls over OpenSSL because a security tool shouldn't carry native C library vulnerabilities. Every tradeoff was made with specific user scenarios and failure modes in mind."

Common questions interviewers might ask about this project, with prepared answers

System Design

Walk me through the overall architecture of ScopeWarden.
ScopeWarden is a 5-layer architecture. At the bottom is the Foundation layer — git repositories, YAML config, and JSON file storage. Above that are the Core Engines: a deterministic policy engine for glob-based pattern matching, a git diff analyzer using libgit2, a multi-provider LLM judge, and a session manager. The CLI layer is a Rust binary using clap for command parsing — it's completely agent-agnostic. The Integration layer provides thin adapter files for each AI agent (Claude, Codex, Gemini, Cursor) in their native config format. Finally, the Consumer layer is the AI agents themselves, which invoke ScopeWarden via shell commands. Each layer only depends on layers below it — clean dependency direction.
Why Rust? What were the alternatives?
Three main reasons: performance (this tool runs on every commit check, so latency matters), memory safety (no garbage collector pauses, no runtime crashes from null pointers), and single-binary distribution (users install one binary — no Python virtualenvs, no node_modules). I considered Go (simpler concurrency model, faster compile times) and Python (larger ecosystem, faster prototyping), but the performance requirements and distribution story made Rust the clear choice. The tradeoff is slower development and a steeper learning curve, but for a security tool that needs to be reliable, Rust's guarantees are worth it.
How does the deterministic policy engine work alongside the LLM judge?
It's a defense-in-depth design. The policy engine runs first — it compiles glob patterns from scopewarden.yaml and evaluates every changed file. Blocked verdicts are final and non-overridable — no LLM can bypass them. This gives us a hard security floor. The LLM judge is an optional layer on top that adds semantic intelligence. For example, if the mission is "fix login bug" and the agent modifies utils/string_helpers.rs, the policy engine might flag it as SUSPICIOUS (not in expected paths), but the LLM judge could determine it's actually relevant because the login code depends on string helpers. The key principle: deterministic rules for security, probabilistic AI for intelligence.
How do you handle the agent-agnostic design? What if a new AI agent comes along?
The core CLI is a pure command-line tool with no knowledge of specific agents. Agent integration is achieved through thin adapter files — each agent has a configuration file in its native format. Claude uses CLAUDE.md + slash commands, Codex uses instructions.md, Gemini uses AGENTS.md, Cursor uses .mdc rules. Adding a new agent literally means creating a new config file — zero Rust code changes. This is the Open/Closed Principle: open for extension, closed for modification. The workflow rules are the same across all agents; only the config format differs.

Technical Deep Dive

How does the data flow when running `scopewarden check`?
10-step pipeline: (1) main.rs parses CLI args via clap. (2) config.rs loads and parses scopewarden.yaml. (3) mission.rs loads the active mission from .scopewarden/mission.json. (4) diff.rs uses git2 to compare HEAD vs working tree — gets Vec<ChangedFile>. (5) policy.rs creates a PolicyEngine, compiling glob patterns. (6) diff.rs classifies each file: policy verdict + mission match → ScopeLabel. Blocked overrides everything. Expected = policy OK + mission match. Suspicious = policy OK + no match. (7) Check limits against max_files_changed, max_lines_changed. (8) Record check result in active session. (9) Output colored report or JSON. (10) Exit code 0 if clean, 1 if blocked files found.
Tell me about your error handling strategy.
I use Rust's Result<T> type with the anyhow crate. Every function returns Result<()> or Result<T> — no panics in normal code paths. Errors propagate with the ? operator for clean bubbling. anyhow::bail!() for early returns with descriptive messages, and anyhow::Context for adding context (e.g., "failed to load config at {path}"). I use distinct exit codes: 0 for success, 1 for policy violations (expected outcome, not an error), 2 for configuration errors (actual errors). This distinction is important — a non-zero exit from scopewarden check doesn't mean the tool failed; it means it found problems, which is its job.
Why did you choose `git2` over shelling out to `git`?
Three reasons: (1) No process spawning overhead — git2 is an in-process library, not a subprocess call. For a tool that might check diffs frequently, this matters. (2) Structured data — git2 gives me typed Rust structs for diffs, file statuses, and tree entries. Parsing git diff output strings is fragile and locale-dependent. (3) Cross-platform reliability — I don't depend on git being on PATH or being a specific version. The tradeoff is a larger binary (libgit2 is a C library linked in) and a C dependency in the build, but for a developer tool that needs to be reliable, I'll take the larger binary.
How would you scale this if the policy engine needed to handle thousands of rules?
Currently, policies are compiled into glob::Pattern instances and evaluated linearly per file. For thousands of rules, I'd consider: (1) Trie-based matching — build a prefix trie from the glob patterns for O(path_length) matching instead of O(num_patterns). (2) Pattern categorization — separate exact matches (hash lookup) from glob patterns (linear scan). Most policies are either exact paths or simple globs. (3) Caching — if the same files are checked repeatedly, cache policy verdicts by file path. (4) Parallel evaluation — use Rayon for data-parallel policy evaluation across files. But honestly, for the current use case (20-800 files, ~20 patterns), the linear approach is microseconds. I'd only optimize when profiling shows it's a bottleneck — premature optimization is the root of all evil.

Leadership & Process

What was the hardest design decision you made?
The deterministic vs. AI-powered split. Early on, I considered making the entire system LLM-powered — just send the mission and the diff to an LLM and ask "is this in scope?" It would have been simpler to build. But I realized that for a security tool, you can't have probabilistic enforcement. An LLM might be convinced through prompt injection that modifying .env is "necessary for the mission." So I split the architecture: deterministic glob-based rules for hard security boundaries, LLM judge for semantic scope assessment. This doubled the complexity but gave us a system where the security floor is guaranteed and the intelligence ceiling is high.
If you were to rewrite this, what would you change?
Three things: (1) Git hooks integration — currently agents run scopewarden check voluntarily. I'd add a pre-commit git hook that automatically blocks commits with BLOCKED files, making enforcement mandatory. (2) Plugin system with dynamic loading — the plugins/ directory exists but isn't populated. I'd implement a proper plugin system with WASM-based plugins for custom policy evaluators, so users can write custom rules without forking the project. (3) Structured logging — currently session data is simple JSON files. I'd add structured event logging (OpenTelemetry-compatible) for better observability in enterprise environments.
How would you explain this project to a non-technical stakeholder?
"AI coding assistants are incredibly powerful, but they sometimes make changes they shouldn't — like modifying security credentials or editing files unrelated to their task. ScopeWarden is like a security guard for your codebase. You tell it what the AI is supposed to be working on, and it watches everything the AI does. If the AI tries to touch a sensitive file, ScopeWarden blocks it immediately. If the AI wanders outside its assigned scope, ScopeWarden flags it for human review. Think of it as guardrails for AI development — you get the productivity benefits of AI while maintaining security and governance."
What metrics would you use to measure ScopeWarden's effectiveness?
Five key metrics: (1) Scope violation rate — percentage of agent sessions with SUSPICIOUS or BLOCKED files (should decrease as agents learn to use ScopeWarden). (2) False positive rate — SUSPICIOUS files that are actually legitimate changes (calibrates mission specificity). (3) Blocked incident rate — how often agents attempt to modify truly sensitive files (security metric). (4) Latency P99 — time for scopewarden check (should stay under 200ms). (5) Adoption rate — percentage of teams/repos using ScopeWarden (platform metric). I'd also track judge accuracy — when the LLM judge says "relevant", how often do human reviewers agree?
